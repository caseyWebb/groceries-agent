# Design — normalization-audit-calibration

## Context

The re-audit passes shipped by `normalization-decision-reaudit` ran their first live sweep and worked as designed structurally, but exposed calibration and id-construction defects. Production facts pinned during planning (remote D1, 2026-07-02):

- Exactly one ≥3-segment node exists: `salmon fillets, skin-on::species-atlantic-sockeye::species-atlantic-sockeye` (auto, surviving, `reconfirmed_at` NULL, `detail` = `species-atlantic-sockeye::species-atlantic-sockeye`). The alias `atlantic sockeye salmon fillets` points at it (stamped), a born-stamped structural edge points from it onto the 2-segment parent, and — critically — the 2-segment parent `salmon fillets, skin-on::species-atlantic-sockeye` has `representative` = **the 3-segment node** (the alias audit's orphan merge ran child-ward). So the repair is not a plain merge: `mergeIdentities(overflow → prefix)` would resolve both to the same root and no-op on the cycle guard.
- 39 `edge_drop` log rows, 23 `edge_keep`; backlog remaining: 201 un-audited aliases, 26 un-audited edges; 307 surviving nodes, 64 edges. The drops split into: structural drops with **live** from-nodes (`rotel (original)::heat-mild→rotel (original)` id 605, `snacking pickles::form-chips→…` id 608, `serrano or jalapeño peppers::form-diced→…` id 598, `grits::cook-time-minute→grits` id 646, `garlic::form-jarred→garlic` id 658); structural drops with **merged-away** from-nodes (`fish sauce::type-sea-salt→fish sauce` id 549, `toasted sesame seeds::toast→toasted sesame seeds` id 556, `salmon fillets, skin-on::form-canned→…` id 552); wrongly-dropped membership/containment (`honey raisins→raisins` id 599, `sweet maui mango habanero sauce→hot sauces (various)` id 602, `bulldog sauce→…` id 601, `jellied cranberry sauce→jellies and jams (various)` id 606, `kale→salad greens` id 650, the korean-ramen-toppings members 652/654/657); the cardamom pair both-deleted (ids 563 + 565, verdict `neither`); correct drops that must stay dropped (`semolina flour→all-purpose flour` 581, the flour family 583/585/613/620, `fruit pectin→jellies and jams (various)` 607, `tuna in oil→tuna in water` 561, `ground nutmeg→whole nutmeg` 568); and one 2-cycle mislabel casualty — `whole frozen chicken -[containment]-> chicken tenderloin` (id 574) was deleted carrying verdict `forward` (the model labeled the direction against its own stated reason, so the good edge died and the bad reverse `chicken tenderloin→whole frozen chicken` was kept + stamped).
- The rejected co-resolution pair is `pecorino romano`/`parmesan` on SKU `0001111050195` — correctly kept distinct, re-confirmed every 5-minute tick (`mergeRejected: 1` in every `ingredient-normalize` job_health row).
- The punctuation duplicate `salmon fillets skin-on` **already self-healed**: its `representative` points at `salmon fillets, skin-on` (the re-confirm pass merged it). Defect D therefore needs prevention only, no cleanup.
- `corpus-db.ts` contains an intentional NUL byte (`\x00` is the pair-key separator inside `filterCommittableEdges`), so it must be patched via script (python/node), not Edit/grep.
- The admin Decisions stream filters `r.outcome.startsWith("edge_")` (`src/normalize-admin.ts` line ~298), so a new `edge_restore` outcome is excluded with no admin change.

## Goals / Non-Goals

**Goals**

- No ≥3-segment id can ever be constructed again, and the one live overflow node converges organically.
- Structural edges (`X::detail → X`, from-node surviving) never cost an LLM call and can never be deleted by the audit; the wrongly-dropped structural class is restored deterministically.
- The recalibrated direction check restores the wrongly-dropped membership/containment class via a one-shot, bounded, self-quiescing log replay.
- The co-resolution pass stops burning one classifier call per tick on a remembered rejection, while still allowing eventual retry.
- Punctuation-variant terms stop minting duplicate nodes.
- Job topology unchanged: everything folds into the three existing jobs; `job_health` summary shapes change additively only.

**Non-Goals**

- No general revisiting of `edge_keep` decisions (a symmetric keep-replay was considered and rejected: the keeps are overwhelmingly correct, and re-selecting the stamped population would reintroduce the destructive-pass risk this capability spent two changes containing). The ONE carve-out, per architect ratification: when the replay of a drop row finds a standing resolved reverse edge, the pair is re-decided by the recalibrated direction check and a wrongly-kept reverse is deleted under that verdict (design D7) — pair-scoped, bounded by the 39-row drop backlog, never a sweep over keeps.
- No id-changing re-home of nodes (unchanged from re-confirm v1); the overflow repair moves `representative` pointers only, plus at most an INSERT OR IGNORE prefix/base mint.
- No suppression of the both-human co-resolution skip (it costs no model call; suppressing operator-pinned pairs behind a table would only hide them).
- No new MCP tools, routes, or admin surfaces.

## Decisions

**D1 — Specialization segment guard lives in `buildResolution`, demoting to SAME.** When `confirm.outcome === "specialization"` and `confirm.match` contains `::`, the resolution is built as SAME on the match (no node mint, edges mapped onto the match id), with log detail `{ note: "specialization_demoted", proposed_detail }`. Rationale: v1 ids are capped at `base::detail`, so ANY specialization of an already-detailed match cannot mint a valid deeper id; SAME-with-match is the conservative resolution (the match already carries the distinguishing detail — exactly the sockeye case). One guard site covers both callers (capture's `resolveOne` and the alias audit's `decide`); the re-confirm pass never constructs ids (verified — it takes only the general-edge subset). `validateCanonicalId` already caps the NOVEL path.

**D2 — Alias-audit no-op keep guard checks the NOVEL canonical before `buildResolution`'s collision fallback.** In `decide()`: when the confirm is NOVEL and `validateCanonicalId(confirm.canonical)` resolves (through the in-tick representative view) to the standing survivor `previous`, return a keep resolution (id = `previous`, no node, detail `{ audit: "alias", previous_id, note: "canonical_is_standing" }`). Without this, `buildResolution` sees the canonical collide with a known id and falls back to minting the VARIANT verbatim — a duplicate node whose only purpose is to shadow the standing mapping. The specialization-demotion path (D1) already lands on `previous` and is counted as kept by the existing `target === previous` check in `auditOne`.

**D3 — Segment-overflow repair folds into the `ingredient-normalize` job as a deterministic sub-pass, with a re-root primitive.** Each tick (before `reconcileCoResolution`), surviving `source='auto'` nodes whose id splits into ≥3 `::` segments are repaired onto their 2-segment prefix (`segments.slice(0,2).join("::")`), three shapes:
1. Prefix node exists and resolves elsewhere → plain `mergeIdentities(overflow, prefix)` (cycle-guard safe).
2. Prefix node exists but resolves TO the overflow node (the live production case) → **re-root**: one batch sets `prefix.representative = NULL` and `overflow.representative = prefix`, logged (`outcome: "merge"`, detail `{ note: "segment_overflow", reroot: true }`). Chain members that pointed through the overflow keep resolving correctly (they end at the overflow, which now points to the prefix).
3. No prefix node → INSERT OR IGNORE the prefix (base/detail derived, `search_term` flattened, `concrete` copied from the overflow, embedding NULL — the existing backfill embeds it) then point the overflow at it, same batch.
A single new corpus-db writer `repairSegmentOverflow(env, plan)` executes whichever shape as one `d.batch`. Human overflow nodes are skipped (counted, never touched). Rationale for folding here rather than a new job: it is node hygiene owned by the pass that mints nodes; it reads only what the pass's siblings already read (`readIdentitySources`); and the steady-state cost is a string filter over ~300 in-memory rows. Downstream convergence is free: the alias resolves through the new chain, `sku-cache-rekey` re-keys any rows, and the now-self-looping born-stamped structural edge is swept by D5(a).

**D4 — Structural exemption is textual shape + a liveness condition.** `isStructuralEdge(from, to)` := `from === to + "::" + <single detail segment>` (from has exactly 2 segments). The audit applies the exemption only when `resolve(from) === from` (the specialization node is surviving): keep + stamp, log `edge_keep` `{ note: "structural" }`, no model call — checked before the reverse-pair branch, and a structural edge is never deleted as the reverse side of a 2-cycle verdict (it is kept + stamped deterministically instead). The liveness condition matters: production holds structural-shaped drops whose from-nodes were merged away (`fish sauce::type-sea-salt`), where the resolved edge would be semantic garbage (sea salt → fish sauce); those fall through to the normal resolved-endpoint logic.

**D5 — Structural guarantee + stamped-self-loop sweep run as a deterministic pre-pass of the edge-audit job, every tick, before the early-return.** `auditEdges` restructures to load identities + edges first and run: (a) **sweep** — delete any `source='auto'` edge whose resolved endpoints coincide, stamped or not (the existing self-loop rule only reaches un-stamped rows; the post-D3 salmon structural edge is stamped), logged `edge_drop` `{ note: "self_loop" }`; (b) **guarantee** — for every surviving 2-segment node `X::d` with no edge of any kind from `X::d` to `X`, INSERT OR IGNORE the `general` edge born-stamped (`audited_at` set), minting the base node `X` (INSERT OR IGNORE, embedding NULL → backfill) when absent, logged `edge_restore` `{ note: "structural_guarantee" }`. Write-capped per tick (20) for safety; idempotent — a converged run plans nothing. Rationale: restores the rotel/pickles/diced class on the first tick with zero model calls, and permanently holds the invariant against future drops or skipped commit-time edges. A prep-detail node destined to be merged away later (e.g. `serrano or jalapeño peppers::form-diced`) may get its structural edge restored first and swept after the merge — both moves are idempotent and the interim edge is a correct hint, so ordering is harmless.

**D6 — Direction-prompt recalibration, not a new validator.** `DIRECTION_SYSTEM_PROMPT` reframes satisfies as "a shopper who has FROM can acceptably fulfill a recipe or shopping request for TO" — explicitly NOT "FROM is the identical product as TO". Membership: a specific product fulfills a request for a category concept it belongs to; the concept never fulfills the member. Whole→derived retained. The distinct-products rule is narrowed to same-level specific products (semolina vs all-purpose, tuna-in-oil vs tuna-in-water, spaghetti vs rigatoni, frozen vs dried fruit) with an explicit instruction not to use it to deny specific→general/category or whole→derived directions. Few-shots: keep the existing four; add `honey raisins→raisins` (forward), `sweet maui mango habanero sauce→hot sauces (various)` (forward, membership), `fruit pectin→jellies and jams (various)` (neither — an ingredient for making the product is not the product), `frozen fruit mix→dried fruit blend` (neither). The output contract (`forward|reverse|both|neither`) and `validateDirection` are unchanged.

**D7 — Replay mechanism: mark rows in log detail (`replayed_at`), born-marked going forward; parse the term string; a standing reverse escalates to a pair re-decision.** Folded into the edge-audit job after the drain, bounded (`EDGE_REPLAY_MAX_PER_TICK = 10`), oldest-first. Selection: `outcome='edge_drop'` rows whose JSON detail lacks `replayed_at` (full read + JS filter, the module's fake-D1-compatible idiom; 39 rows total). Per row:
- Parse `from`/`kind`/`to` with the strict pattern `/^(.+) -\[(general|containment|membership)\]-> (.+)$/` (unparseable → mark `{ replay: "unparseable" }`, no model call).
- Deterministic drops (`detail.note` of `self_loop`/`human_reverse`) → mark, no model call. Structural rows with a surviving from-node → mark `{ replay: "structural" }` (D5's guarantee restores the edge; replay does not double-write). From-endpoint missing or merged away, or resolved endpoints coincide → mark `{ replay: "endpoint_merged" }`.
- Otherwise, when NO resolved reverse edge exists: one recalibrated `checkDirection(readable(resolve(from)), readable(resolve(to)))` — verdict holds (`forward|both`) → re-insert the ORIGINAL endpoints (INSERT OR IGNORE, source auto, born-stamped `audited_at`) and append an `edge_restore` log row with `{ audit: "edge", replay_of: <log id>, direction, reason }` plus structured `from`/`to`/`kind`; verdict doesn't hold → mark with the verdict. Then mark the source row `{ replayed_at, replay, direction?, reason? }`.
- **When a resolved reverse edge DOES exist, escalate to a pair re-decision (architect-ratified amendment — never withhold a restore behind a standing reverse).** Deterministic guards first: a `source='human'` reverse wins with no model call (mark `{ replay: "human_reverse_standing" }`, no restore — human authority, audit parity); a structural reverse (D4 shape with a surviving from-node) is definitionally valid, is never deleted, and blocks the restore (mark `{ replay: "structural_reverse" }`, no model call). Otherwise the SAME single `checkDirection` call decides the whole pair under the audit's 2-cycle semantics: `forward` → restore the dropped edge AND delete the standing reverse (logged `edge_drop` with `{ note: "replay_cycle", replay_of: <log id> }`, born-marked, structured detail); `both` → restore the dropped edge, keep the reverse; `reverse` → mark `{ replay: "stands" }`, reverse untouched; `neither` → mark `{ replay: "stands" }` AND delete the reverse (neither direction holds). Still one model call per escalated row. The deleted reverse may carry an earlier `edge_keep` stamp — this pair-scoped carve-out is deliberately the one place the change revisits a keep, and it is what converges the production mislabel casualty: the replay of row 574 restores `whole frozen chicken -[containment]-> chicken tenderloin` and deletes the wrongly-kept `chicken tenderloin -[general]-> whole frozen chicken`.
- Transient error → row left unmarked (unmarked IS the retry state); contract-invalid → mark `{ replay: "confirm_failed_safe" }`, no restore (fail safe = don't resurrect on an undecidable).
Post-change `edge_drop` writes (all sites) are **born-marked** (`replayed_at` set at write time — they are already under recalibrated judgment), and all edge log rows gain structured `from`/`to`/`kind` detail fields so any future replay never parses strings. Quiesce: ~4 ticks drain the 39-row backlog, then the reader returns nothing. Why detail-marking over a table or a job-state flag: it is per-row (idempotent under partial ticks and restarts), keeps the audit trail self-describing (the verdict lands beside the drop it re-judged), and avoids a second migration; a single done-flag was rejected because a partial tick would either redo LLM calls or lose rows.

**D8 — Rejection memory is a D1 table (`0036_coresolution_rejections.sql`) keyed on ordered surviving ids, with a 30-day backoff.** `ingredient_coresolution_rejection(a, b, decided_at, PRIMARY KEY (a, b))` where `(a, b)` is the pair's surviving ids at decision time, lexicographically ordered. `reconcileCoResolution` filters candidate pairs whose key has `now - decided_at < NORMALIZE_CORESOLVE_REJECT_BACKOFF_MS` (30 days), counting them as `merge_suppressed`; a confirm rejection upserts the row (refreshing `decided_at` on a post-backoff re-rejection). Chosen over log-mining because it survives restarts trivially, reads in one small query, and gives refresh semantics without JSON surgery; chosen over a registry-version salt because the survivor-keyed pair already invalidates naturally — any later merge changes a survivor id, changing the key, making the pair immediately eligible again (exactly "the graph changed materially"). Transient confirm failures do NOT record a rejection (they stay `mergeSkipped`, retried next tick). Both-human skips are not recorded (no model cost; see Non-Goals).

**D9 — Defect D in scope: deterministic punctuation-equality fast path + one prompt rule.** `lexicalKey(s)` = lowercase, every character outside `[a-z0-9:]` collapsed to single spaces, trimmed (`:` survives so `base::detail` ids keep their shape). Capture (`resolveOne`, before the floor check) and the alias audit (`auditOne`, before the confirm) look the term up in a map of `lexicalKey(id) → survivor` over surviving node ids plus `lexicalKey(variant) → survivor` over alias variants; a unique hit resolves SAME deterministically (`{ note: "lexical_match" }`, no embedding comparison, no model call); an ambiguous key (two distinct survivors) skips the fast path. The identity-confirm prompt gains the rule + few-shot: a term differing from a candidate only in punctuation, pluralization, or word order is SAME (`'salmon fillets skin-on'` vs candidate `salmon fillets, skin-on` @0.98 → same). Pluralization/word-order folding is deliberately NOT in the deterministic path (false-positive risk; the LLM rule covers it). Rationale for keeping D in scope: the production instance cost a wasted mint plus a healing re-confirm call, prevention is ~40 lines, and the fast path also removes a whole class of alias-audit confirms. No cleanup component — production already converged.

**D10 — Fold, don't add jobs.** All new behavior lives inside the three existing Phase-1 jobs (`ingredient-normalize`, `ingredient-alias-audit`, `ingredient-edge-audit`); `src/index.ts` needs at most comment updates (no wiring change, no new `Promise.allSettled` entries, `HEALTH_JOBS` untouched). Summary shapes extend additively: `NormalizeSummary` += `segment_repaired`, `merge_suppressed`, `lexical`; `EdgeAuditSummary` += `structural` (deterministic keeps), `structural_restored`, `self_loops_swept`, `replayed`, `restored`. `AliasAuditSummary` unchanged (lexical/no-op keeps count under `kept`/`repointed`).

## Risks / Trade-offs

- **Recalibration could over-restore** (satisfies too lax → noise edges). Mitigations: the conservative rule and the true-drop few-shots stay; replay is one-shot, bounded, contradiction-gated, and fully logged (`edge_restore` rows carry `replay_of`); edges remain open-world hints — a wrong edge degrades to read-time judgment, never a wrong purchase. Some borderline drops (the shredded-cheese membership class, `poblano→whole green chilies`, `dried→canned chickpeas`) are left to the recalibrated model's judgment either way.
- **The re-root primitive writes `representative` pointers outside `mergeIdentities`' choke point.** Contained: it runs only for surviving auto ≥3-segment nodes (production count: exactly 1; the guard prevents new ones), the exact production shape is a test fixture, and the write is a single two-statement batch that cannot create a cycle (the prefix becomes a root; the overflow points at it).
- **Replay's pair escalation can delete a stamped keep.** Scoped tightly: only the reverse of a replayed drop row, only under a fresh recalibrated verdict, with human and structural reverses deterministically immune, and every deletion logged (`replay_cycle`, `replay_of`) and born-marked. This is bounded by the drop backlog (39 rows, one-shot), so it cannot become an ongoing destructive channel; the general `edge_keep` population is still never re-selected.
- **Steady-state cost**: the edge-audit job now does two full-table reads (~300 + ~64 rows) every tick even when quiesced, matching `sku-cache-rekey`'s existing pattern; negligible at this scale.
- **Detail-JSON mutation** (`markEdgeDropReplayed` rewrites a log row's detail): the log stops being strictly append-only for these rows. Accepted deliberately — the mark is additive inside the row, preserves the original fields, and keeps the audit self-describing.

## Migration Plan

1. Migration `0036_coresolution_rejections.sql` applies with the deploy (`wrangler d1 migrations apply --remote`), before the Worker rollout as usual; the table starts empty and the first post-deploy tick records the pecorino/parmesan rejection once.
2. Everything else converges organically: tick 1 restores the structural class (D5) and repairs the salmon overflow (D3); ticks 1–4 drain the replay backlog (D7); the co-resolution summary drops to `mergeRejected: 0` steady-state with `merge_suppressed: 1`.
3. Rollback is a Worker redeploy of the previous version; the new table and marked log rows are inert to old code (old code never reads `replayed_at` or the rejection table). Restored edges remain — they are ordinary born-stamped auto edges.
4. Post-deploy acceptance (remote D1, no writes): the queries listed in tasks 8.3.
