# Design — recipe-capture-funnel

## Context

The ingredient-identity graph (`organic-ingredient-normalization` + follow-ups) is fed by capture-on-encounter through the one `IngredientContext` funnel: grocery/pantry writes, the matcher, the meal-plan proposal's set math, and the classify pass all `resolve*` through it, which normalizes AND best-effort-enqueues misses to `novel_ingredient_terms`. The recipe corpus is the hole: the classify pass is the corpus's only funnel pass, it is change-gated (`facetGateHash` over body + Tier-B overrides), and the funnel was wired in after the corpus was first faceted — so 259 distinct legacy terms (681 of 949 stored keys in production) are stored in `recipe_facets` as cleaned-but-unplaced strings and never enqueue. The projection (`reconcileRecipeIndex`) copies them verbatim into `recipes` each tick via `loadClassifiedFacets` → `mergeEffectiveFacets` → `recipeToRow`, never touching the funnel.

Two distinct symptoms, one root: (1) the graph never learns the corpus's vocabulary, so recipe-side joins (cross-recipe overlap in `use-it-up`/`meal-plan-proposal-tool`, grocery-list dedup against recipe needs, pantry matching) miss; (2) projected ids are frozen at classify-time resolution, so a later graph improvement (the decision re-audit re-pointing a bad alias, a synonym merge) does not reach the index until a body edit re-classifies the recipe.

Constraints: the projection must stay deterministic plain code — no LLM, no embedding calls (capture is enqueue-only; the normalize cron disposes); the resolver read must not become a new projection failure mode; `recipe_facets` is the classify pass's table and the projection must not write it; no D1 schema change.

## Goals / Non-Goals

**Goals:**
- The projected `recipes.ingredients_key`/`perishable_ingredients` carry **current** surviving canonical ids on every tick.
- Every unplaced term in the projected corpus is enqueued for capture on every tick until the graph places it — backfilling the legacy corpus and self-healing future capture gaps with no manual backfill.
- A resolver read failure degrades (project everything, cleaned passthrough, capture off) exactly like the grocery/pantry write paths.
- The operator can observe convergence (an `unresolved` gauge in the job summary).

**Non-Goals:**
- No rewrite of `recipe_facets` values (the classify pass owns that table; its rows stay as-classified snapshots).
- No reclassification trigger — recipes are NOT re-run through the classifier to fix resolution; re-resolution is a pure map over stored values.
- No change to the classify pass's own funnel call (it remains the capture point for newly-classified content).
- No new tool or admin surface; no migration.

## Decisions

- **Re-resolution happens in `reconcileRecipeIndex`, after `mergeEffectiveFacets`, via `resolveNames`.** The context is loaded once per pass through a new `ProjectionDeps.ingredientContext(): Promise<{ ctx: IngredientContext; degraded: boolean }>` (alongside `loadClassifiedFacets`) — a resolve-ONLY context (capture off) — and the two effective arrays are mapped through `ctx.resolveNames(...)` before `recipeToRow`. Post-merge is the right hook: it covers both the classified values AND the pre-migration authored Tier-A fallbacks (`mergeEffectiveFacets` falls back to authored frontmatter for an unclassified recipe), and it changes only the projected row, never the stored facet. `resolveNames` over `resolveList` because the effective values are already validated `string[]` (`ClassifiedFacets`-typed) — the lenient always-`string[]` set-builder is the exact shape `recipeToRow` needs, it dedupes/drops empties. Capture is the projection's own single batched flush: the pass collects its distinct unplaced ids and enqueues them ONCE via `enqueueNovelTerms` after the loop — skipped on a degraded pass, awaited so the write is not orphaned past the invocation, `.catch`-guarded so a flush failure never fails the pass — so a term shared by 40 recipes enqueues once and the pass performs one queue write instead of one per term.
- **Projected ids are the AFTER-representative-chain surviving ids.** The funnel offers no "before" — `readResolver` bakes the representative chain into the variant→id map, so `resolveNames` only ever emits survivors; storing pre-chain ids would mean open-coding a chain-less resolver against the one-funnel rule. It is also the only correct join value: the `ingredient-normalization` spec's full-canonical-id join rule keys everything on the id *after* synonym-merge, and every comparison partner is resolved fresh at read time (`use-it-up` resolves pantry names via `resolveNames`; `meal-plan-proposal-tool` resolves boost/pantry/perishable-vocab names the same way) — a pre-chain recipe-side id would miss those joins for every merged node. Staleness is bounded by one tick: the projection re-runs wholesale each cron, so a merge landing between ticks re-points the whole index on the next one. `perishable_ingredients` follows identically — it feeds the same consumers (the perishable-vocab union, use-it-up overlap) with the same join semantics; diverging the two fields would just split the invariant.
- **Capture is not food-gated.** The food guard (`isFoodItem(kind, domain)`) exists to keep non-food *grocery rows* (household/other kind, non-grocery domain) out of the graph. Recipe ingredient facets have no kind/domain and are food by construction — they are derived from the recipe body's `## Ingredients` section by the facet classifier, exactly the vocabulary the graph exists to hold — the same wholesale-funnel treatment pantry already gets. Gating would add a judgment call with no input signal to judge on.
- **Degradation is the standard empty-context fallback, wired in `buildProjectionDeps`, plus an explicit `degraded` marker.** `ingredientContext(env, { capture: false })` falling back to `emptyIngredientContext(env)` with `degraded: true` — the grocery/pantry-write degradation pattern with the failure made legible. On a resolver read failure the pass still projects every recipe: `resolveNames` degrades to lowercase/strip passthrough of the stored values (which are already cleaned, so the projected rows equal today's verbatim copy) and the capture flush is skipped (an unreadable resolver must not flood the queue with terms it cannot check). The projection's own failure semantics (skip-and-record per recipe, throw only on job-level failure) are untouched.
- **`unresolved` (distinct count) and the `degraded` flag join `ProjectionResult` and the `recipe-index` job summary (`{ projected, skipped, unresolved, degraded }`); the usage-trends doubles do not change.** Counting resolved ids not in `ctx.resolver.ids` across projected recipes is free at projection time and is the convergence gauge (949-key corpus: watch 259 → 0 over ~11 normalize ticks). It rides `job_health`/`job_runs.summary` (a free-form JSON object; SCHEMAS.md documents the shape). `recordUsagePoint` keeps `[projected, skipped]` — the AE series' double positions are consumed positionally by the trends views, and the gauge belongs to liveness detail, not the long-horizon trend. A degraded (empty-context) tick reports every term unresolved AND sets `degraded: true` — the explicit alertable marker for the resolver-read failure, which is otherwise invisible because the projection deliberately succeeds; the job stays ok (it genuinely projected).
- **Wholesale-rebuild interplay: none to add.** Re-resolution is a pure per-row map inside the same build → single-transaction replace; `recipe_facets` survives the rebuild untouched exactly as the existing requirement demands. The projection's determinism statement widens: the table is a deterministic function of (R2 corpus, `recipe_facets`, resolver state) — still plain code, still replayable.
- **Cost: one resolver read + O(keys) map lookups per tick.** The resolver read is two full-table SELECTs over small shared tables (identities + aliases, low hundreds of rows), same as every hot-path tool request already pays; the map work is ~184 recipes × ~5 keys of dictionary lookups. The one-time first-tick enqueue burst is one batched insert-or-ignore flush of ≤ 259 distinct ids, thereafter ≈ 0 as the graph converges. No LLM, no embeddings, no external subrequests.

## Risks / Trade-offs

- [Projected `recipes` values drift from the `recipe_facets` rows they came from] → Intended and now the documented semantics: `recipe_facets` is the classify-time snapshot, `recipes` is the join surface at current resolution. SCHEMAS.md states it so an operator diffing the two tables isn't surprised.
- [A merged-away id stored in `recipe_facets` that has no alias-variant row re-enqueues as "novel"] → `resolveNames` misses it (no variant row) and `captureIfNovel` sees it absent from the survivor set, so it enqueues; the capture job then disposes it — worst case one cheap classifier confirm ending in SAME (an alias onto the survivor), after which it resolves forever. Self-healing, bounded, and rare (mints write the surface term as an alias variant, so most ids re-point through the map).
- [First-tick enqueue burst (≤ 259 distinct ids in one invocation)] → One batched insert-or-ignore `enqueueNovelTerms` flush per pass (a single D1 batch), best-effort by construction (it swallows its own errors and the call is `.catch`-guarded); a failed flush leaves the terms unresolved, so the next tick re-flushes them. The burst decays to zero as the graph converges.
- [The captured forms are the classify-time cleaned strings, not the recipes' raw surface text] → Exactly what the classify pass's own funnel would have enqueued for these recipes had it existed then; the queue's contract is cleaned surface forms.
- [A degraded tick reports a scary `unresolved` spike] → That spike is the only signal of a resolver-read failure on this job (the projection deliberately still succeeds); treating it as signal, not noise, is the point.

## Migration Plan

Code-only; no D1 migration. Deploys with the normal Worker deploy; the first post-deploy cron tick begins enqueueing the legacy backlog and the normalize job drains it at its existing 25/tick pace (~11 ticks). Rollback = revert the commit — projected rows fall back to verbatim facet copies on the next tick (the wholesale rebuild leaves nothing to clean up), and already-captured graph nodes are simply kept (they are ordinary capture output).
