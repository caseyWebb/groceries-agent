## Context

The Usage area (`src/admin/pages/usage.tsx` + `src/usage.ts`) is the original thin implementation: three `status-row` lists rendering `fetchUsage` (KV op totals + per-namespace totals for the **current UTC day only**, plus Workers AI neurons, via the Cloudflare GraphQL Analytics API), `fetchUsageTrends` (per-job runs/day + avg duration over 30 days, via the Analytics Engine SQL API over the Worker-owned `grocery_usage` dataset), and `fetchToolUsage` (per-tool calls/errors/p50/p95 over 30 days, via AE SQL over `grocery_tool`). The foundation change already landed the kit primitives this composes from (`StatCardGrid`/`StatCard`, `Progress`, `Sparkline`, `DataTable`) and a global health dock; this change is presentation plus the one real backend gap the mock's design exposes.

The mock (`UsageScreen.jsx`, `usage-data.jsx`) wants, per KV operation (read/write/delete/list): a meter bar **stacked by namespace** (a categorical color per namespace) against the daily limit, plus a **30-day sparkline also stacked by namespace** with a per-namespace tooltip breakdown on hover. It labels namespaces by friendly id (`KROGER_KV`, `OAUTH_KV`, `TENANT_KV`) with a note (`tokens · flyer cache`, `oauth grants · tokens`, `directory · invites`).

**What already exists:** `mapAccountUsage` already groups the GraphQL `kvOperationsAdaptiveGroups` rows by `namespaceId` for **today**, producing `usage.kv.namespaces: NamespaceUsage[]` (id + per-action counts). The existing comment in `src/usage.ts` is explicit about the one hard limitation: *"a Worker cannot map ids back to binding names at runtime"* — the GraphQL Analytics API reports Cloudflare's opaque per-namespace `namespaceId` (a 32-char hex string), not the `wrangler.jsonc` binding name. There is no Workers runtime API that resolves a `KVNamespace` binding to its underlying namespace id (the binding is just a handle), so the Worker genuinely cannot self-discover "this id is KROGER_KV" — that mapping can only come from outside the request path (an operator-supplied config value, or a one-time lookup against the Cloudflare REST API using a token with namespace-list scope, run out-of-band).

**What does NOT exist:** a **history** of the per-namespace breakdown. Today's snapshot is one GraphQL query filtered to a single `date`; there is no 30-day per-namespace series anywhere.

## Goals / Non-Goals

**Goals:**
- Redesign the Usage area's presentation (stat tiles, per-namespace-stacked KV meters, AI neuron meter + per-model chips, per-job sparklines, tool-usage table) using existing kit primitives.
- Decide, with an honest cost/accuracy accounting, how to source a **30-day per-namespace KV history** — the one real data gap the mock's design requires and today's backend does not provide.
- Decide how a Cloudflare namespace id becomes a friendly label (`KROGER_KV`/`OAUTH_KV`/`TENANT_KV`) and a stable display color, given the Worker cannot resolve this itself at request time.

**Non-Goals:**
- No change to the **AI-neuron** or **per-job**/**per-tool** trends data paths — `fetchUsageTrends`/`fetchToolUsage` already provide 30-day history at the granularity the mock wants; this change only composes their existing output into the new presentation.
- No new Worker-side KV-operation instrumentation (i.e. the Worker does **not** start writing its own KV-op-count data points to a custom AE dataset) unless Decision 1 concludes the GraphQL-history approach is insufficient — see Decision 1's "Option B" for why that's the fallback, not the default.
- No change to the daily free-tier limits the meters measure against (`FREE_TIER_LIMITS` in `src/usage.ts`).
- No redesign of other areas (Status, Members, Data, Discovery, Logs, Config) — out of scope.

## Decisions

### 1. Per-namespace 30-day KV history: extend the existing GraphQL query to a date range (recommended), not a new AE instrumentation path

Three realistic options, in the order the investigation considered them:

**Option A — widen the existing GraphQL Analytics query to a 30-day date range, grouped by `(namespaceId, actionType, date)`.** `kvOperationsAdaptiveGroups` is filtered today by `date_geq: "<day>", date_leq: "<day>"` (today only) with `dimensions { namespaceId actionType }`. Cloudflare's GraphQL Analytics API is a managed, Cloudflare-populated dataset — not something the Worker writes to — and it already supports filtering by an inclusive date *range* with `dimensions` extended to include `date` (the same shape `kvOperationsAdaptiveGroups` already exposes elsewhere in Cloudflare's own dashboards, which chart KV ops over arbitrary date ranges). Widening the existing query is therefore: change the filter to `date_geq: "<30 days ago>", date_leq: "<today>"`, add `date` to `dimensions`, and reshape `mapAccountUsage`'s grouping (or a sibling `mapAccountUsageHistory`) to bucket by `(namespaceId, actionType, date)` instead of just `(namespaceId, actionType)`. **Cost: zero new KV/D1, zero new Worker-side writes, one GraphQL query already being made just widened** — the same "observing the budget must not consume it" property the snapshot already guarantees, extended to history. The query's row cap (`limit: 1000`) comfortably covers `namespaces(~3) × actions(4) × days(30) ≈ 360` rows, well under the API's per-query row ceiling.

**Option B — instrument KV ops into a custom Analytics Engine dataset (the `usage-trends`/`tool-usage-trends` pattern), tagged with a namespace label.** This would mean wrapping every `KVNamespace` access (`KROGER_KV.get`, `TENANT_KV.put`, etc.) to emit a `writeDataPoint` per operation, the same registration-seam pattern `tool-instrumentation.ts` uses for MCP tool calls. **Rejected as the default**: it is strictly more expensive and more invasive than Option A for the same outcome — it adds a wrap to *every* KV call site (a much hotter, more numerous seam than the once-per-tool-call wrap, since a single tool can issue many KV ops), it duplicates data Cloudflare's own GraphQL Analytics already collects for free, and `writeDataPoint` itself is non-blocking but not literally free (CPU + a write-path call on every KV operation, where Option A spends nothing on the hot path at all). It would only be justified if Option A's data turns out to be unavailable, insufflently granular, or too short a retention window in practice — see the validation step below.

**Option C — approximate from the daily snapshot by sampling it once per day into KV or D1.** Take today's `mapAccountUsage` snapshot and persist it (e.g. a small D1 `kv_usage_daily` table, one row per `(namespace_id, action, day)`) on a cron tick, building up history operator-side instead of querying Cloudflare for it. **Rejected as the default**: it requires a new D1 table, a new cron job (or piggy-backing an existing one), and — ironically for a *usage*-observability feature — consumes real D1 write budget to record KV usage, plus it only starts accumulating history from the day it ships (no backfill), whereas Option A can show the full prior 30 days immediately because Cloudflare already has that data. It is the right move if and only if Option A's GraphQL range query turns out not to exist or not to retain 30 days for this dataset (see below) — at which point it degrades gracefully to "shorter history" rather than "no history."

**Recommendation: Option A**, with two flagged honesty caveats the apply step must verify against the live API (the existing `src/usage.ts` header already carries this discipline — *"Cloudflare evolves these dataset/field names; they are verified against the live schema as part of landing this capability"* — and this extension inherits it):
- **Retention**: the GraphQL Analytics API's retention window for `kvOperationsAdaptiveGroups` is not independently confirmed in this repo (unlike the AE SQL datasets, where `docs/SCHEMAS.md` already notes "AE free-tier retention ≈90 days" from direct experience). It is reasonable to expect KV analytics retention to comfortably exceed 30 days (Cloudflare's dashboard UI itself offers 30/90-day views over this exact dataset), but the apply step's live-schema verification (already a stated task discipline for this file) MUST confirm a 30-day range query actually returns 30 days of rows, not a truncated window — if it doesn't, fall back to Option C for the uncovered tail, never silently render a short series as if it were the full 30 days.
- **Row-cap arithmetic holds only while the namespace count stays small.** The `limit: 1000` ceiling is comfortable for 3 namespaces × 4 actions × 30 days ≈ 360 rows; this is a friend-group-scale Worker with exactly three KV bindings, so it is not expected to grow, but the query SHOULD log (or surface in the not-configured/degraded state) if the response appears truncated at the row cap, rather than silently rendering a partial picture as complete.

**This is the area's main backend cost, and it is honestly cheap, not expensive** — the per-namespace breakdown already exists for today; widening it to 30 days is a query-shape change, not a new data-collection mechanism. The thing that is **not** cheap, and is irreducible, is namespace **labeling** (Decision 2) — that's the part with a real operator-facing cost.

### 2. Namespace id → label/color: an operator-supplied static mapping, not a runtime lookup

Because the Worker cannot resolve a `namespaceId` to its binding name at request time (no such runtime API exists — confirmed by the existing code comment), and a server-side lookup against Cloudflare's REST API (`GET /accounts/<id>/storage/kv/namespaces`, which *does* return id→title pairs) would mean a **second** Cloudflare API surface, an extra outbound call on every Usage-page read (or a cache with its own staleness/invalidation story), and a broader token scope (namespace listing, beyond the read-only Analytics scope `CF_ANALYTICS_TOKEN` already carries) — this change recommends a small **static, operator-configured mapping** instead:

- A non-secret var (e.g. `KV_NAMESPACE_LABELS`, JSON or `id:LABEL,id:LABEL` pairs) the operator fills in once after `wrangler kv namespace create` reports each namespace's id (the same moment they already paste KV ids into `wrangler.jsonc` per `docs/SELF_HOSTING.md`'s "Persisting your namespace + database ids" step) — or, more conveniently, the Worker can read the ids already present in its own `wrangler.jsonc`-resolved binding config at deploy time (the `kv_namespaces[].id` values the operator pins) and a code-level constant mapping `{ KROGER_KV: "tokens · flyer cache", OAUTH_KV: "oauth grants · tokens", TENANT_KV: "directory · invites" }` keyed by **binding name**, joined against the deploy's own pinned ids — see Open Questions for which of these two is simpler to wire without a new operator-facing var.
- An id with **no** label mapping (a namespace the Analytics API reports that isn't one of the three known bindings — e.g. before the operator has pinned an id, or a future fourth namespace) renders as a generic "unlabeled" segment (its raw id, muted/gray) rather than being dropped — so the totals/limits stay accurate even when labeling is incomplete, and an operator who hasn't configured the mapping yet still sees correct aggregate meters, just without the per-namespace color breakdown.
- Colors are a small fixed categorical palette (3–4 swatches) assigned by binding name in code, not operator-configurable — consistent with the mock's `--kv-kroger`/`--kv-oauth`/`--kv-tenant` CSS custom properties.

**Alternative considered**: a one-time Cloudflare REST lookup (`storage/kv/namespaces`) cached in KV. Rejected — it would itself consume a KV read/write to cache the very ids that are usually pinned once into `wrangler.jsonc` at provisioning time and never change again; using data the deploy already has (the binding→id pin) is strictly simpler and consumes zero ongoing budget.

### 3. Per-namespace meter + sparkline rendering: pure SSR, hover detail via native `title`, no island

The mock's `Spark`/`KvMeter` components use a JS-driven custom tooltip (`useTip`) for hover detail (per-namespace breakdown on a sparkline bar). Per `src/admin/CLAUDE.md` rule 8 ("a page that only *reads* is pure SSR... no island, no client fetch"), the Usage area has no mutation and no genuine interactivity — it is a read-only dashboard like the existing implementation (whose own "Refresh" is a page reload, not an island). This change keeps it that way: the per-segment/per-bar hover detail is carried by a native `title` attribute (and/or a `<details>`/CSS-only affordance) on each stacked segment and sparkline bar, not a JS tooltip component — consistent with the kit's `Sparkline` primitive already being presentational-only with no hover behavior of its own. A future change MAY add a richer hover-tooltip island primitive to the kit if another area needs one; this change does not introduce one for Usage alone.

### 4. `UsageResult.kv` shape: add a `history` series alongside the existing snapshot, not a replacement

`UsageResult.kv` keeps its existing `limits`/`totals`/`namespaces` (today's snapshot, unchanged — still drives the `MeterRow`-equivalent total-vs-limit bar) and gains a new field, e.g. `kv.history: { window_days: number; days: { day: string; namespaces: { namespace_id: string; read: number; write: number; delete: number; list: number }[] }[] }` — one entry per day, oldest→newest, mirroring the `JobTrend`/`ToolUsage` series shape already established by `usage-trends`/`tool-usage-trends` (ascending by day, `YYYY-MM-DD`). This keeps the discriminated-union discipline: `UsageResult` stays `{ configured: false } | { configured: true; ... }`, with `history` only present in the `configured: true` branch — no new top-level endpoint or route, since the same `CF_ACCOUNT_ID`/`CF_ANALYTICS_TOKEN` config gates both the snapshot and the history (one query, or two GraphQL queries from the same `fetchUsage` call — see Open Questions).

## Risks / Trade-offs

- **[Risk]** The GraphQL Analytics API's actual retention/row behavior for a 30-day ranged `kvOperationsAdaptiveGroups` query is unverified against the live schema (the existing file already flags this evolving-schema risk for the same dataset's same-day query). → **Mitigation**: apply step verifies against a live account before relying on it; if retention is short or the range filter behaves unexpectedly, fall back to Option C (D1-accumulated daily snapshots) for the shortfall rather than rendering a misleadingly short series as the full window.
- **[Risk]** Without an operator-configured (or deploy-pinned) namespace label mapping, the per-namespace meter degrades to "unlabeled" generic segments — less useful than the mock's labeled/colored design. → **Mitigation**: ship the deploy-time binding→id join (Decision 2's preferred sub-option) so a correctly-provisioned operator gets labels with zero extra config; document the manual `KV_NAMESPACE_LABELS` fallback for the case where ids aren't resolvable at build/deploy time.
- **[Trade-off]** Widening the GraphQL query to 30 days returns a chattier response (~360 rows vs. ~12 today) on every Usage-page load. This is well within the API's row cap and the page is operator-only/low-traffic, but the response is parsed and reshaped server-side on every SSR — acceptable at this scale, not free at a hypothetical larger one (irrelevant for a friend-group-scale Worker).
- **[Risk]** A future change adding a 4th KV namespace silently renders as "unlabeled" until the operator/deploy config catches up. → **Mitigation**: the unlabeled-segment fallback (Decision 2) means this degrades gracefully (accurate totals, generic color) rather than breaking the page.

## Migration Plan

No data migration: this is a read-only observability surface widening an existing outbound query, plus an optional new non-secret config var (or a deploy-time join requiring no new var). No D1 schema change unless Option C's fallback is triggered by the apply step's live-API verification, in which case a new `kv_usage_daily` migration would be added at that time — not anticipated as the primary path. No feature flag needed; the page degrades exactly like the existing not-configured states when `CF_ACCOUNT_ID`/`CF_ANALYTICS_TOKEN` is unset.

## Open Questions

- Does the live Cloudflare GraphQL Analytics API actually return a full 30-day range for `kvOperationsAdaptiveGroups` grouped by `(namespaceId, actionType, date)`, or does it cap/truncate sooner? Apply step must verify against a live, configured account before committing to Option A as final (per the existing file's stated "verified against the live schema" discipline).
- Decision 2's two labeling sub-options — operator-pasted `KV_NAMESPACE_LABELS` var vs. a deploy-time join against the `kv_namespaces[].id` values already pinned in `wrangler.jsonc` — which is simpler to wire without adding new operator setup steps? Leaning toward the deploy-time join (zero new operator action for anyone who already followed the existing "persist your namespace ids" step), but apply should confirm the Worker can read its own resolved binding ids at runtime (vs. only at deploy-merge time) before committing.
- Should `fetchUsage` issue the snapshot and the 30-day history as one widened GraphQL query (date range covering today, with "today" derived as the max `date` in the result) or two separate queries (today's snapshot kept exactly as-is, history added as a second query)? The latter is a smaller, lower-risk diff to the already-tested `mapAccountUsage`; the former is fewer outbound requests. Recommend the latter (additive, not a rewrite of the working snapshot path) unless apply finds a compelling reason otherwise.
