## Goals / Non-Goals

**Goals:** every shim named in `docs/TOOLS.md`'s meal-dimension removal-condition paragraph is gone from the tool/route surface; the three frozen `profile` columns are dropped; the cadence read-fallback is a fixed default with no legacy escape hatch; no code is left querying a column the migration just dropped.

**Non-Goals:** the `close-cull-windows` fusion aliases and the `single-slot-discovery` `new_for_me` window â€” different changes, already closing on their own schedule. Any behavior change to cadence or meal-vibe semantics beyond removing the named shims. A Claude Design pass on the member app's profile/propose pages â€” the app-side edits this change needs are mechanical source swaps (read `cadence.dinner` instead of a mirror field), not UI redesign.

## Decisions

- **Column drops are plain `ALTER TABLE ... DROP COLUMN`, not a table rebuild.** The codebase's own precedent for narrowing a table with no primary-key change is a direct drop: `0012_overlay_favorites_rejections.sql` (`ALTER TABLE overlay DROP COLUMN status`; `DROP COLUMN rating`; `ALTER TABLE ready_to_eat DROP COLUMN status`) and `0013_recipe_derived.sql` (`ALTER TABLE recipes DROP COLUMN description`). Table rebuilds in this codebase (`0052_meal_dimension`'s `meal_plan` rebuild) are reserved for primary-key changes, which dropping three nullable scalar columns doesn't require. D1's SQLite (workerd, `compatibility_date: 2025-06-01`) supports `ALTER TABLE DROP COLUMN` natively for ordinary columns. *Alternative rejected:* CREATE-new/INSERT-SELECT/DROP/RENAME â€” unnecessary ceremony inconsistent with the codebase's own two precedents for this exact operation. The migration:

  ```sql
  ALTER TABLE profile DROP COLUMN default_cooking_nights;
  ALTER TABLE profile DROP COLUMN lunch_strategy;
  ALTER TABLE profile DROP COLUMN ready_to_eat_default_action;
  ```

- **The `pref-retirement` pass is deleted outright, not left registered as a no-op.** Found by survey, not named in the original chartered scope: `packages/worker/src/index.ts:23,336` imports and calls `runPrefRetirementSeedJob(env)` inside the `scheduled()` handler's phase-5 `Promise.allSettled` batch on every cron tick. The job's own `SELECT` (`packages/worker/src/pref-retirement.ts`) names `lunch_strategy` and `ready_to_eat_default_action` literally. Once the migration drops those columns, that query throws `no such column` on every future tick. `Promise.allSettled` keeps that from aborting its phase-5 siblings, but `scheduled()`'s final line (`index.ts:339-340`) still finds the rejected settlement and re-throws â€” so every cron invocation would report failed, forever, for a pass whose own docstring already promises termination once converged (verified, tonight). *Alternative rejected:* leave it registered as a permanent no-op â€” its SELECT can't no-op against a dropped column; it errors. Deleting it is required, not a style choice.
- **`packages/worker/test/pref-retirement.test.ts` is deleted with it**, and `packages/worker/test/meal-dimension-migration.test.ts` is confirmed untouched: it applies migrations only through `0052` via its own `applyBefore(raw, "0052_meal_dimension.sql")` helper, never the full chain, so it never sees the new `0063` drop and keeps testing `0052`'s already-shipped backfill behavior unmodified.
- **`effectiveCadenceCount`'s fallback becomes a bare `5`.** Behavior is unchanged for every caller whose `cadence.dinner` is unset today (the fallback value was always `5`); only the legacy `default_cooking_nights ?? 5` escape hatch is dropped, since the column it reads no longer exists.
- **`RETIRED_PREFERENCE_KEYS` and `applyRetiredKeyShim` (`packages/worker/src/preferences.ts`) are deleted wholesale, not emptied-but-kept.** Once the shim's two branches (retired-key accept-and-drop; `default_cooking_nights` alias-and-merge) are gone, both keys fall through to the same generic `rejectUnknownPatchKeys` unknown-key rejection for free â€” neither key was ever in `DEFINED_PREFERENCE_KEYS`. There's no bespoke rejection behavior worth preserving in a permanently-empty conditional. This mirrors `close-cull-windows`' own precedent of deleting shim branches rather than leaving inert scaffolding.
- **Two small, pre-existing spec/code drifts are corrected in the same sentences this change already rewrites, rather than left behind:** `data-write-tools`' "Preferences are edited by merge-patch..." requirement lists `update_preferences`'s "defined top-level surface" as `default_cooking_nights, planning_cadence_days, lunch_strategy, ready_to_eat_default_action, weekly_budget, stores, brands, dietary, rotation, custom` â€” nine names that already don't match `DEFINED_PREFERENCE_KEYS` in code (`cadence, planning_cadence_days, weekly_budget, stores, brands, dietary, rotation, curated_hide, custom`), predating this change (no `cadence`, no `curated_hide`, and treating the retired trio as "defined" when they were only ever handled by the now-closing shim). Since this change is rewriting that exact list to drop the retired trio, it also adds `cadence` and `curated_hide` so the sentence matches shipped code. This is the only unrelated-seeming fix in this change, and it is confined to the one list already being edited â€” not a broader `data-write-tools` audit.
- **The mcp-tool-gating requirement's *other* in-flight parenthetical (`remove-ready-to-eat`) is left untouched**, even though that change is already archived and the reference looks stale. That staleness predates this change and belongs to a different change's cleanup debt; fixing it would be scope creep. Only the `remove-meal-dimension-shims` clause â€” this change's own â€” is removed here.
- **`packages/ui/src/propose-controller.ts`'s `defaultNights` option keeps its name.** It's an internal controller-option name ("the default dinner count to seed a blank session with"), not a wire-protocol field â€” only its caller (`packages/app/src/routes/_app.propose.tsx:53-54`) changes which preference it reads to compute the value it passes in (`prefs.cadence?.dinner` in place of `prefs.default_cooking_nights`). Renaming the option itself would be gratuitous churn on the shared controller's public surface for no behavior change.
- **`PROPOSE_SESSION_VERSION` (`packages/ui/src/propose-orchestration.ts`) bumps 4 â†’ 5.** `ProposeSession`'s persisted shape drops the `nights` field, matching the module's own stated contract: "Bumped when the shape changes so a stale localStorage blob is discarded rather than mis-read."
- **The `VarietyBar`/`ProposeCard` `nights` **prop** is unrelated and stays untouched.** `packages/ui/src/components/propose.tsx`'s `VarietyBar({ nights })` and `packages/widgets/src/ProposeCard.tsx`'s `nights={controller.summary.filled}` render a "N nights planned" display stat sourced from the plan's filled-slot count â€” not the wire `nights` alias or `diagnostics.nights`. ## Risks / Trade-offs

- [A production profile row carries a non-NULL `default_cooking_nights` post-migration for an undiscovered reason] â†’ the verified query found only 2 of 4 rows with a non-NULL value, and both also carry authoritative `cadence` (read-time precedence already ignores the mirror); dropping the column loses only an already-inert frozen scalar, never the effective cadence.
- [A stale plugin bundle still calls `add_night_vibe`, sends `nights`, or hits `/vibes/suggest`] â†’ three-member deployment, operator-assisted (the same waiver posture `close-cull-windows` established); failures are structured unknown-tool / unknown-key / 404 rejections, never silent misbehavior.
- [The `pref-retirement` removal is a bigger surface than the chartered scope named] â†’ verified via its `SELECT` literally naming the columns this migration drops (`packages/worker/src/index.ts:23,336`, `pref-retirement.ts`); leaving it would fail every cron tick against the dropped columns, so removing it is required once the migration ships, not optional scope creep.
- [`packages/contract` edits trip the satellite-version CI gate for a package that didn't otherwise change] â†’ expected: the gate (`â€‹.github/workflows/ci.yml`'s `satellite-version` job) covers `packages/contract/**` regardless of whether `packages/satellite` code moved. Bump `packages/satellite/package.json`'s version in the same PR.
- [A widget or member-app client session persisted in localStorage under the old shape] â†’ `PROPOSE_SESSION_VERSION` bumps (4 â†’ 5); a stale persisted session fails the version guard on load and is discarded, falling back to a fresh default session â€” no crash, no corrupted read.

## Migration Plan

The D1 migration and the code that stops selecting the three dropped columns ship in the same deploy. Wrangler applies pending D1 migrations before the new Worker version receives traffic, so there is no window where the previous (column-selecting) code runs against the already-migrated (column-dropped) schema. The new code never references the dropped columns, so it is correct regardless of exactly when the migration lands relative to the version cutover.

Rollback = revert the deploy. The code reverts; the dropped D1 columns do not un-drop automatically (D1 migrations aren't auto-reverted) â€” a hypothetical rollback would need a follow-up migration re-adding the columns (nullable, no historical values recoverable) before the old code could run again. Given verified convergence (nothing effective was reading the frozen values), this mirrors `0012`'s own "PURE SUBTRACTION... irreversible" framing: a deliberate, verified-safe one-way cleanup, not a defect needing a safety net.
