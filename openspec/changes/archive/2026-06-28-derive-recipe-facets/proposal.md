## Why

Recipe facets like `protein`, `cuisine`, `course`, `ingredients_key`, and `perishable_ingredients` live in authored R2 frontmatter today, yet most are not genuinely authored — they are AI-classified once (in chat at import, or by the discovery sweep's `src/discovery-classify.ts` on the cron) and then frozen into the file. That violates the system's own placement rule (*"a field lives in frontmatter only if a human authors or corrects it; a purely-derived, regenerable field lives in D1"* — the rule that already moved `description` to `recipe_derived`). The consequences: a classifier improvement never reaches existing recipes, the same facets get classified twice by two diverging code paths, and an author of a body-only recipe must still hand-fill metadata a model could infer.

The guiding principle this change applies: **humans author the hard gates and the identity; the AI derives the descriptive facets.** The fields that stay required in frontmatter become exactly the two hard gates (`dietary`, `requires_equipment`) plus identity (`title`, `source`).

## What Changes

- **Add a whole-corpus classify pass** to the `scheduled()` handler — a generalization of the existing discovery classifier (`src/discovery-classify.ts`, mistral-small-3.1-24b, contract-validated with corrective retry) over the entire R2 corpus. It is `body_hash`-gated (reclassify only when a body changes → steady-state ≈ 0 work, mirroring the `description` `content_hash` gate), reads R2 bodies directly (off the 50-external-subrequest budget), and runs **before** the recipe-index projection.
- **BREAKING (authoring/corpus): Tier A facets become derived-only**, removed from authored frontmatter — `ingredients_key`, `perishable_ingredients`, `side_search_terms`, and `meal_preppable`. `meal_preppable` does not exist in the code today (only a doc example); it is **implemented new** on the classifier as a boolean.
- **Tier B facets become optional in frontmatter** with override semantics — `protein`, `cuisine`, `course`, `season`, `tags`. Absent → the classifier fills D1; present → the authored value wins. `tags` is the one exception: **union** semantics (`effective = classified ∪ authored`), so an editorial tag never wipes the classified set.
- **Tier C is untouched** — `dietary` and `requires_equipment` (the hard gates) plus `title`, `source`, `time_total`, and the authored numerics stay required/authored in frontmatter. This deliberately preserves human authority over the two facets whose misclassification has real consequences (allergen exposure; silently hiding a makeable recipe).
- **The projection materializes effective facets into the `recipes` index** (`effective = authored ?? classified`; union for `tags`), so every existing reader — `filterRecipes`, the cookbook, the retrospective, the describe pass, semantic search — keeps reading `recipes` columns unchanged.
- **The required-field contract shrinks** (`src/recipe-contract.js`): Tier A leaves it, Tier B becomes optional-but-validated-when-present, Tier C stays required; the `course` → `side_search_terms` cross-rule moves to classifier-output validation (already enforced there).
- **The authoring vault slims** — the recipe fileClass and "New recipe" template shed the Tier A fields and demote Tier B to optional-override dropdowns; `requires_equipment`'s equipment-vocab dropdown stays load-bearing.
- **One-time strip-on-agreement migration**, driven by an agreement eval (the change's first task): run the classifier over the current authored corpus (existing frontmatter facets are free ground-truth labels), strip Tier A unconditionally, and for Tier B strip where the classifier agrees with the authored value while keeping the disagreements as authored overrides — so the surviving frontmatter is precisely the human corrections.
- **`create_recipe` stops requiring the agent to supply A/B facets** and seeds classification synchronously (as it already seeds `description`), keeping agent imports index-lag-free; `update_recipe` still accepts B-overrides and the Tier C fields.

## Capabilities

### New Capabilities
- `recipe-facet-derivation`: the whole-corpus cron classify pass that derives the Tier A (pure) and Tier B (override-default) recipe facets from the recipe body, `body_hash`-gated and bounded per tick, materialized into the recipe index; plus the strip-on-agreement migration and its agreement eval.

### Modified Capabilities
- `recipe-metadata-contract`: Tier A fields leave the required-field contract; Tier B fields become optional-when-absent and validated-against-vocab-when-present; Tier C (`dietary`, `requires_equipment`) stay required; the `course` → `side_search_terms` conditional moves to classifier-output validation.
- `recipe-index`: the projection merges effective facets (`authored ?? classified`, `tags` unioned) into the `recipes` table from the classifier's sibling data; the index becomes a function of body-derived facets plus authored overrides, not authored frontmatter alone.
- `recipe-import`: `create_recipe` no longer requires the agent to supply Tier A/B facets and seeds classification synchronously; `update_recipe` edits Tier B as overrides and Tier C as authored fields.
- `recipe-authoring-vault`: the recipe fileClass + "New recipe" template slim to identity + gates + body, with Tier B as optional-override dropdowns and Tier A removed.
- `derived-recipe-metadata`: the describe pass's content-hash requirement is generalized — the facets it hashes are now the recipe's *indexed* facets (authored-or-derived), read from the D1 index, while the describe pass itself still does not read the body (the new classify pass is the only body-reading derive step); `meal_preppable` ceases to be the named free-form-passthrough example (it becomes derived).

## Impact

- **Code:** new shared classify module extracted from `src/discovery-classify.ts` (consumed by both the new whole-corpus pass and the discovery sweep); `src/recipe-projection.ts` (effective-facet merge); `src/recipe-contract.js` + `src/validate.ts` (contract shrink); `src/recipe-embeddings.ts` (the `content_hash` cascade — derived facets now drive description/embedding regeneration); `src/discovery-tools.ts` + `src/write-tools.ts` (`create_recipe`/`update_recipe`); `scripts/build-vault.mjs` + `vault-template/` (slimmed authoring surface); `src/discovery.ts`/`src/discovery-sweep.ts` (reuse the shared classifier + alias normalization of `ingredients_key`/`perishable_ingredients`).
- **Data / migration:** a new D1 table (or extension of `recipe_derived`) for classified facets + change-detection hash — decided in design; a one-time R2 corpus rewrite (strip-on-agreement); a whole-corpus reclassify → re-describe → re-embed on first run (bounded per tick).
- **Crons:** `scheduled()` ordering changes (classify before projection); the classify pass draws on the internal `env.AI` budget and R2 body reads, alongside the existing flyer/projection/derived/discovery jobs; a `health:job` record for the new pass.
- **Latency:** a direct Obsidian/R2 body-only edit is un-faceted until the next classify tick (eventual-consistency, acceptable); the `create_recipe` path stays lag-free via the synchronous seed.
- **Docs (lockstep):** `docs/SCHEMAS.md` (frontmatter, `recipes` projection, the facet table), `docs/ARCHITECTURE.md` (the new pass + placement-rule restatement), `docs/TOOLS.md` (`create_recipe`/`update_recipe`), `CLAUDE.md` (vault description).
- **Out of scope:** `dietary` and `requires_equipment` stay authored; the discovery sweep's pipeline (triage/dedup/match/gate) is unchanged — but its import *write* now strips the descriptive facets and seeds `recipe_facets` (the create_recipe model), so discovered recipes are derived, not re-frozen.
