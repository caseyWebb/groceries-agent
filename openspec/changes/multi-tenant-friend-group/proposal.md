## Why

The system is single-tenant by construction: one repo's coordinates + PAT in `env`, one Kroger refresh token at a fixed KV key, and Cloudflare Access gating the Worker by team membership for the one user. The goal is to let **the technical member of a friend group self-host one instance for the whole group** — without each friend standing up their own Worker (constraint) or registering their own Kroger Developer app (constraint). A research pass confirmed the Kroger public tier's shared-app + per-user-consent model is viable at friend-group scale: the read grant (`client_credentials`) is app-level, the cart grant (`authorization_code` + `cart.basic:write`) is per consenting user, and the 5,000 cart-calls/day per-app cap is far above a handful of users (the one unread risk is the public-tier Acceptable-Use clause on serving non-owner users — noted, low blast radius at this scale).

Multiplexing users onto one instance also unlocks something a personal tool can't: with privacy moot inside a trusted group, the recipe corpus becomes **shared and collaborative** — "three of your friends rated this 4+, want to try it?" — and personal spins on a shared recipe are captured as **attributed notes** rather than forks. The system becomes one collaborative cookbook with personal lenses.

## What Changes

- **The Worker becomes a multi-tenant OAuth 2.1 provider.** Claude.ai custom connectors require OAuth (no bearer/no-auth option), so each friend connects their own Claude.ai to the shared Worker; the issued access token resolves to a tenant. Identity is gated by a **curated allowlist** the operator maintains (not open registration). **BREAKING:** Cloudflare Access edge-gating is removed for the MCP surface.
- **Per-request tenant context replaces global repo coordinates.** `GITHUB_OWNER/REPO/REF` + a static PAT stop being global; the Worker resolves the caller's tenant, loads that tenant's repo coordinates, and mints a short-lived **GitHub App installation token** scoped to the right repo. No per-user PAT, no per-user Worker secret.
- **Model B repository layout.** One operator-owned org holds a **shared corpus repo** (objective recipe content + reference data + indexes) plus **one private repo per friend** (personal state). A single GitHub App install on the org grants the Worker write access to all of them. Friends own zero infrastructure; repos are transferable out if someone leaves.
- **Three-category data model.** Recipe data splits along the seam the frontmatter already has: **content** (objective, shared — title/ingredients/time/cuisine/components), **overlay** (per-tenant single value — `rating`, `last_cooked`, `status`), and **notes** (shared, multi-author, attributed, append-mostly). Read tools join shared content + the caller's overlay; a private-recipe escape hatch lets a friend keep personal recipes unshared.
- **Recipe notes (new, must-build).** Attributed notes authored in a friend's own repo and surfaced across the group; a per-note `private` flag keeps a note personal. Notes are how personal spins are captured without forking shared content — which is what makes the shared corpus safe to share.
- **Shared SKU cache + reference data.** `skus/kroger.toml`, `aliases.toml`, `ingredients.toml`, `substitutions.toml` move to the shared corpus. SKU entries become **location-tagged**; the existing per-hit revalidation against the caller's `preferred_location` already makes a shared cache correct, with a network-effect win (one friend's resolution warms everyone's).
- **Per-tenant Kroger authorization.** Each friend runs the one-time Kroger consent against the operator's shared app; the refresh token is stored per-tenant (`kroger:refresh:<tenant>`), PKCE `state` binds to the tenant, and the access-token cache becomes per-tenant (the current module singleton is a correctness bug under multi-tenancy).
- **Self-hosting guide (new).** `docs/SELF_HOSTING.md` — the operator's one-time setup (register the GitHub App + Kroger app, create the org + shared-corpus repo, set OAuth-provider secrets) and the per-friend onboarding runbook. Doubles as the operator's own reference.

**Non-Goals:** open public signup or a hosted "free instance" (this is self-host-for-friends only); a SQL database (domain data stays in flat files in repos; only operational mapping — tenant→repo+install, tenant→Kroger token, OAuth grants — lives in KV); cross-tenant privacy controls beyond the per-note `private` flag (the group is trusted by stipulation); automating each friend's Claude.ai Project instruction paste (a Claude.ai platform limit — documented in the setup guide).

## Capabilities

### New Capabilities
- `multi-tenancy`: tenant identity via the Worker as an OAuth 2.1 provider gated by a curated allowlist; per-request tenant resolution from the bearer token; per-tenant repo coordinates; GitHub App installation token minting for repo writes; per-tenant Kroger refresh-token storage; the Model B org/repo model.
- `shared-corpus`: the content/overlay split — a shared recipe corpus (objective content + frontmatter) joined at read time with the caller's per-tenant overlay (`rating`/`last_cooked`/`status`, default `draft` when absent); the private-recipe escape hatch (read = shared ∪ personal); shared SKU cache (location-tagged) + reference data.
- `recipe-notes`: attributed, append-mostly notes authored in a tenant's repo and surfaced across the group; per-note `private` flag; cross-tenant aggregation at read time.

### Modified Capabilities
- `mcp-server`: the Cloudflare Access edge/JWT gate is replaced by per-request OAuth bearer → tenant resolution; the MCP server is built per `(env, tenant)` rather than per `env` alone.
- `kroger-user-auth`: the refresh token and PKCE `state` are keyed per tenant; the access-token cache is per-tenant (no module singleton); each friend authorizes their own Kroger account against the shared app.
- `repo-structure`: Model B layout — one shared corpus repo + one private per-tenant repo under the operator's org; per-tenant `overlay/` and `notes/` directories; reference data + SKU cache relocated to the shared corpus.
- `data-read-tools`: recipe reads/filters join shared content + the caller's overlay (and personal recipes); notes are surfaced aggregated across the group.
- `data-write-tools`: writes are routed to the correct repo (shared corpus vs. the caller's per-tenant repo); overlay and note writes are introduced.
- `data-indexing`: indexes are built over the shared corpus; per-tenant overlay is merged at read time rather than baked into the shared index.

## Impact

- **Worker code:** new `tenant.ts` (resolve bearer→tenant, load repo coords, mint GitHub App install token); new OAuth-provider surface (authorize/token/registration via `workers-oauth-provider`) + allowlist identity gate; `env.ts` (drop per-repo coords + PAT, add GitHub App id/private key; Kroger read creds stay global); `index.ts` (remove Access gate, add bearer→tenant); delete `access.ts`; `github.ts`/`gh-read.ts`/`commit.ts` (take tenant context); `kroger-user.ts` (`REFRESH_KEY` → per-tenant, per-tenant cache); `oauth.ts` (bind `state`→tenant); `tools.ts` + every `*-tools.ts` (`buildServer(env)` → `buildServer(env, tenant)`); read/write tools (overlay + notes join/routing); matching (location-tagged shared SKU cache).
- **Infra:** one GitHub App registration; one operator org + a template-based shared-corpus repo + per-friend repos; OAuth-provider secrets; a KV namespace for tenant directory + OAuth grants (alongside the existing Kroger KV).
- **Data/repo:** shared corpus repo (recipe content, reference data, SKU cache, indexes) extracted from today's single repo; per-tenant repos carry pantry/preferences/taste/diet_principles/grocery_list/stockup/overlay/notes.
- **Docs:** new `docs/SELF_HOSTING.md`; `docs/PROJECT.md` (architecture: multi-tenant, Model B, three-category data model), `docs/SCHEMAS.md` (overlay + note schemas, location-tagged SKU entries), `docs/TOOLS.md` (read/write tool contract changes, note tools), `AGENT_INSTRUCTIONS.md` (surface group ratings/notes; capture spins as notes; per-tenant connect/consent setup).
- **Relationship to in-flight work:** `cooking-log-and-retrospection` (active, 0/26) overlaps — a note often arises from logging a cook. Open design question (resolved in design.md): whether a cooking-log entry carries notes or notes are a separate capability the log references.
